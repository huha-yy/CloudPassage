# 面试时怎么讲这个项目的回答模板

## 一、30 秒版本

这个项目是一个基于 `Spring AI Alibaba` 的多 Agent 智能图文创作平台。我主要做的是把文章生成流程拆成多个 Agent 节点，比如标题生成、大纲生成、正文生成、正文评审、图片策略路由、配图生成和图文合并。除了跑通主流程，我还重点补了任务快照、节点日志、单节点重放、长期记忆上下文和调试面板，让这个项目更像一个可调试、可恢复、可持续迭代的 Agent 工程系统，而不只是简单调用大模型接口。

## 二、1 分钟版本

这个项目的目标是做一个 AI 图文创作平台，用户输入主题后，系统会自动生成标题、大纲、正文和配图。技术上我用的是 `Spring Boot + Spring AI Alibaba + Vue 3 + TypeScript + Redis + MySQL`。  
我把后端流程拆成多个 Agent 节点，并通过 `SSE` 把执行进度实时推送到前端。为了提高工程可用性，我做了三类增强：  
第一类是可恢复能力，比如任务快照、阶段恢复和单节点重放；  
第二类是可观测能力，比如节点日志、节点时间线、调试面板；  
第三类是记忆能力，比如任务记忆、用户偏好记忆和长期记忆上下文，并把这些记忆接入图片 Router 和正文 Reviewer，让历史经验能影响后续决策。  
所以这个项目的重点不是“接一个大模型”，而是把多 Agent 系统做成一个更接近真实工程的形态。

## 三、2-3 分钟展开版本

如果展开讲，这个项目主要有四块：

### 1. 我解决的核心问题

很多 AI 项目只能演示一次生成结果，但出了问题很难定位，也很难从中间继续执行。所以我这个项目重点解决三个问题：

- AI 工作流怎么拆成多个可维护节点
- 失败后怎么恢复，而不是整条链路重跑
- 历史经验怎么沉淀下来，并参与后续决策

### 2. 我具体做了什么

我把文章生成拆成标题、大纲、正文、Reviewer、图片 Router、图片生成和图文合并等节点。  
然后围绕这些节点做了：

- `SSE` 实时进度推送
- Redis 任务快照，支持阶段恢复
- 节点日志和节点级重放快照
- 单节点重试 / 单节点继续执行
- 任务记忆、用户偏好记忆、长期记忆上下文
- 前端调试台，展示节点时间线、失败原因和执行摘要

后面我还把长期记忆真正接到后端决策里，比如图片策略 Router 会参考历史成功和失败案例，正文 Reviewer 会参考历史质量信号和失败提示。

### 3. 这个项目的难点

我觉得最难的不是调用模型，而是让系统具备工程可用性。

比如：

- 节点重放不能只是在前端点一个按钮，后端必须真的能从某个节点恢复上下文继续跑
- 任务快照和任务记忆不能混在一起，快照负责恢复现场，记忆负责沉淀经验
- 长期记忆不能只展示出来，要真正接入 Router 和 Reviewer，否则就没有实际价值

### 4. 我从这个项目里学到了什么

我最大的收获是，Agent 开发不只是 prompt 和模型调用，更重要的是：

- 工作流拆分
- 状态管理
- 失败恢复
- 执行观测
- 记忆上下文设计

这些能力决定了一个 AI 系统能不能持续开发和维护。

## 四、如果面试官问“你个人贡献是什么”

你可以这样答：

我主要负责的是 Java 主后端和整体 Agent 工程能力增强，重点包括：

- 多 Agent 工作流梳理与后端执行链路整理
- 任务快照、阶段恢复、单节点重放能力实现
- 节点日志、执行观测字段和调试面板后端接口设计
- 图片策略 Router、正文 Reviewer、长期记忆上下文的接入
- Docker 本地部署链路和前后端联调支持

如果你确实也参与了前端，可以再补一句：

前端侧我也参与了调试台和任务观测界面的联调与展示优化。

## 五、如果面试官问“为什么这个项目像 Agent，而不是普通 AI 应用”

回答模板：

我认为它像 Agent，主要不是因为它用了大模型，而是因为它具备了 Agent 系统更关键的几个特征：

- 有多节点分工，而不是单轮问答
- 有 Router 和 Reviewer，能做判断和自检
- 有任务记忆和长期记忆，能复用历史经验
- 有节点重放和恢复机制，能从局部继续执行
- 有调试台和执行观测，便于持续优化

如果只是输入一句话、输出一段文本，我更倾向叫它 AI 应用；但这个项目已经具备比较明显的 Agent 工程特征。

## 六、如果面试官问“项目里最有技术含量的点是什么”

回答模板：

我觉得最有技术含量的点有两个：

第一是单节点重放。因为它不是简单重新跑任务，而是要恢复到某个节点前后的真实上下文，再从这个节点继续往后执行。这里要处理快照恢复、状态重置、事件链路和最终落库一致性。

第二是记忆架构。因为快照、节点日志、任务记忆和长期记忆的职责不能混。快照解决恢复，日志解决观测，记忆解决经验复用。只有边界清晰，后面 Router 和 Reviewer 才能稳定读取上下文。

## 七、如果面试官问“这个项目还有什么可以继续优化”

回答模板：

我觉得后续最值得继续做的方向有四个：

- 把长期记忆继续接入更多节点，比如图片生成和失败恢复策略
- 给节点重放增加输入覆盖能力，支持真正的调试实验
- 建立评测基线，让 prompt、模型和路由策略调整可以量化对比
- 继续把调试台做完整，前后端统一展示 memory context 和 replay context

## 八、回答时的注意点

- 不要上来就说“我做了一个 AI 写作网站”，太普通
- 要强调“多 Agent + 可恢复 + 可观测 + 记忆增强”
- 不要吹成“自研大模型”或“底层 Agent 框架”
- 面试里尽量多讲你怎么解决失败恢复、状态管理和调试问题，这些最能体现工程能力
