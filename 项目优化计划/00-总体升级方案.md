# 项目总体升级方案

## 文档定位

这份文档用于说明项目为什么要升级、优先从哪些方面入手，以及整体升级路线怎么拆分。它是后续各阶段实施文档的上位说明。

## 升级目标

本次升级不追求一次性把功能做得更多，而是先把现有“选题 -> 标题 -> 大纲 -> 正文 -> 配图 -> 合成”的主链路做得更稳定、更清晰、更容易维护。核心目标如下：

- 提升多 Agent 编排链路的稳定性，减少结构化输出失败
- 让任务状态可追踪、可恢复，支持前后端协同恢复
- 统一 SSE 协议与创作页状态管理，降低前后端耦合
- 为后续记忆、路由、评审 Agent、工具策略升级打基础
- 让项目更适合作为长期迭代作品，而不只是可演示 Demo

## 当前现状概述

### 后端现状

- Java 主后端是单体 Spring Boot 架构，文章生成主流程清晰。
- 文章生成当前有两套执行模式：
  - `ArticleAgentService` 旧模式
  - `ArticleAgentOrchestrator` 编排模式
- 多 Agent 雏形已经具备，包含标题、大纲、正文、配图分析、并行配图、图文合成。
- SSE 和异步执行已打通，但消息协议和状态组织仍然偏分散。

### 前端现状

- Vue 3 + TypeScript 主链路完整，核心页面集中在创作页和详情页。
- `ArticleCreatePage.vue` 目前承担了页面展示、SSE 消息解析、阶段切换、日志和进度维护等多重职责。
- 页面能跑通，但随着升级继续叠加功能，维护成本会快速上升。

## 当前主要问题

- LLM 输出高度依赖直接返回 JSON，缺少统一校验与自动重试。
- 任务运行态主要依赖内存对象，缺少标准化快照与恢复机制。
- 前后端依赖字符串事件驱动流程，协议不够统一。
- 图片工具层已经具备雏形，但工具选择、降级和记录能力还不系统。
- 后端双轨执行模式并存，日志与观测能力不一致。

## 总体升级路线

### 第一阶段：稳定性底座

目标是先把现有主链路做稳。

- 建立结构化输出执行器
- 增加任务快照与前后端恢复能力
- 统一 SSE 协议
- 重构创作页状态流转

### 第二阶段：执行内核收敛

目标是让多 Agent 编排真正成为主执行模式。

- 统一编排入口
- 统一日志与节点观测
- 收敛旧模式与新模式之间的职责差异

### 第三阶段：Agent 能力增强

目标是在稳定基础上提升系统智能度。

- 引入任务级记忆与用户偏好记忆
- 增加路由 / Planner 能力
- 增加 Reviewer / Refiner 闭环
- 升级工具路由与降级策略
