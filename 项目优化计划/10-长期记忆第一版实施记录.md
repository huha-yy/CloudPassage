# 长期记忆第一版实施记录

## 本次目标

这一版先不引入向量库和复杂 RAG，而是先打通三件事：

- 长期记忆可以写入
- 长期记忆可以被召回
- 召回结果会真实影响 Agent 决策，而不是只做展示

本阶段重点围绕正文评审、图片路由、图片需求分析、图片生成四个关键节点展开。

## 已完成内容

### 1. 统一记忆上下文模型

已新增：

- `src/main/java/com/yupi/template/model/vo/ArticleMemoryContextVO.java`

作用：统一聚合当前任务记忆、用户偏好、历史成功案例、历史失败案例、图片偏好方法、图片规避方法、质量提示、失败提示，供各个 Agent 节点直接消费。

### 2. 长期记忆召回入口

已补齐：

- `src/main/java/com/yupi/template/service/ArticleMemoryService.java`
- `src/main/java/com/yupi/template/service/impl/ArticleMemoryServiceImpl.java`

新增核心方法：

- `buildCreationMemoryContext(taskId, userId)`

当前召回策略：

- 仅召回当前用户自己的历史任务
- 区分成功任务与失败任务
- 根据风格、主题、图片方法等信息做轻量匹配
- 最多返回 3 条成功案例和 3 条失败案例

### 3. 图片策略路由开始使用长期记忆

已接入：

- `src/main/java/com/yupi/template/service/impl/ImageStrategyRouterServiceImpl.java`

当前效果：

- 当当前状态没有明确方法时，优先参考历史偏好方法
- 会规避历史失败案例中反复出错的方法
- 决策来源可标记为 `long_term_memory`

### 4. 正文 Reviewer 使用长期记忆

已接入：

- `src/main/java/com/yupi/template/service/ArticleAgentService.java`

当前 `agent3_review_content` 会读取：

- 质量提示 `qualityHints`
- 失败提示 `failureHints`
- 相似成功案例摘要
- 相似失败案例摘要

同时节点观测元数据已增强：

- `decisionSource`
- `decisionReason`
- `decisionSummary`
- `memoryContextSummary`
- `memoryContextSnapshot`

### 5. 节点重放保留记忆上下文快照

已接入：

- `src/main/java/com/yupi/template/model/vo/NodeExecutionMetadata.java`
- `src/main/java/com/yupi/template/model/vo/NodeReplaySnapshotVO.java`
- `src/main/java/com/yupi/template/service/impl/ArticleNodeReplayServiceImpl.java`

当前 replay 快照会同步保存：

- `memoryContextSummary`
- `memoryContextSnapshot`

这意味着节点重放不再只是看结果，还能解释当时为什么这样决策。

### 6. 图片生成链路补成可解释

已接入：

- `src/main/java/com/yupi/template/service/ArticleAgentService.java`

当前能力：

- `image_strategy_router` 会记录图片场景的记忆上下文摘要与快照
- `agent5_generate_images` 会记录图片生成场景的记忆上下文摘要与快照
- 图片生成节点会额外记录生成数量、降级次数、典型降级链路

### 7. 图片需求分析节点接入记忆上下文

本次新增：

- `src/main/java/com/yupi/template/service/ArticleAgentService.java`

当前 `agent4_analyze_image_requirements` 已补齐：

- 读取统一记忆上下文
- 写入图片分析场景的 `memoryContextSummary`
- 写入图片分析场景的 `memoryContextSnapshot`
- 写入节点级 `decisionSummary`

当前摘要会记录：

- 本次识别出的配图需求数量
- 当前启用的图片方法数量
- 首个需求使用的图片来源类型

这样图片链路已经形成：

- `image_strategy_router`
- `agent4_analyze_image_requirements`
- `agent5_generate_images`

三段都能解释“参考了什么记忆、做了什么决定、最终走了什么结果”。

### 8. 调试查询入口

已新增接口：

- `GET /article/memory-context/{taskId}`

用途：查看指定任务在执行期实际聚合出的长期记忆上下文，便于后续接入前端调试面板。

## 当前边界

这一版仍然是结构化长期记忆，暂不包含：

- 向量检索
- 跨用户共享经验
- LLM 自动压缩记忆
- 自动失败策略学习闭环

## 下一步建议

优先继续两项：

1. 前端节点调试面板展示 `memoryContextSummary` 与 `memoryContextSnapshot`
2. 把图片降级与图片生成链路进一步沉淀成可复用的 Agent 工具能力
