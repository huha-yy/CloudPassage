# 第一阶段实施方案

## 阶段目标

第一阶段只做“稳定性底座”，不追求增加新功能上限。目标是把 Java 主后端与前端创作页整理成一套更稳定、可恢复、可继续扩展的闭环。

本阶段完成后，应达到以下效果：

- 标题、大纲、配图需求等结构化输出失败率明显下降
- 文章任务状态可以保存并在页面刷新后恢复展示
- SSE 协议统一，前端不再过度依赖零散字符串判断流程
- 创作页状态管理比当前更清晰，为后续继续拆分打基础

## 实施内容

### 1. 后端结构化输出执行器

围绕标题生成、大纲生成、配图需求分析、AI 修改大纲，补一层统一执行逻辑：

- Prompt 调用
- JSON 解析
- 结果校验
- 失败重试
- 标准错误输出

建议首批校验内容：

- 标题数量、主副标题是否为空
- 大纲 section、title、points 是否完整
- 配图需求是否包含 `position`、`type`、`imageSource`
- 图片来源是否在当前允许列表中

默认重试策略：

- 最多重试 2 次
- 第二次提示模型严格返回合法 JSON
- 连续失败后返回标准节点错误，不静默吞掉异常

### 2. 后端任务快照与恢复能力

在保留 `ArticleState` 作为运行态对象的基础上，增加任务快照存储。

快照至少记录：

- `taskId`
- 当前 `phase`
- 已完成节点
- 标题候选
- 已确认标题
- 大纲当前版本
- 已生成正文片段
- 配图需求
- 已生成图片
- `fullContent`
- 最近错误、重试次数、更新时间

保存时机：

- 关键节点完成后保存
- 正文 / 大纲流式输出按节流保存
- 失败时保存失败状态和错误原因

恢复目标：

- 用户刷新页面后可以根据 `taskId` 恢复当前阶段
- 如果任务已完成，能恢复完整内容
- 如果任务失败，能恢复失败状态和失败原因

### 3. SSE 协议统一

统一文章生成 SSE 消息结构，避免前端继续依赖零散前缀解析。

建议统一字段：

- `type`
- `phase`
- `progress`
- `payload`
- `timestamp`
- `taskId`

本阶段要求：

- 阶段切换事件标准化
- 流式正文 / 大纲事件标准化
- 图片完成事件标准化
- 任务完成 / 失败事件标准化

### 4. 创作页状态管理重构

不先改视觉风格，优先改状态组织方式。

建议拆分：

- SSE 连接与消息解析
- 创作阶段状态机
- 任务恢复初始化
- 实时日志维护
- 图片进度计算

页面组件本身主要负责：

- 输入态展示
- 标题选择展示
- 大纲编辑展示
- 正文生成展示
- 完成态展示

## 接口与数据变化

第一阶段默认新增但不大改现有文章接口：

- 保留创建文章、确认标题、确认大纲接口
- 新增任务快照查询接口，用于创作页恢复
- 扩展 SSE 返回对象结构
- 运行态 `ArticleState` 可增加恢复字段，但持久化快照建议使用独立模型

## 验收标准

### 后端验收

- 标题 / 大纲 / 配图需求异常 JSON 能触发重试
- 两次失败后能输出明确错误
- 任务处理中可以查到当前快照
- 任务完成后可以查到完整快照
- 任务失败后可以查到失败 phase 与错误原因

### 前端验收

- 正常创建文章，流程完整可用
- 刷新后能恢复到：
  - 标题选择态
  - 大纲编辑态
  - 正文生成态
  - 完成态
- 正文生成中刷新不丢已生成内容
- 失败任务能展示失败信息并提供重新创建入口

### 手工场景

- 正常生成一篇带配图文章
- 标题确认后刷新页面
- 大纲编辑中刷新页面
- 正文生成中刷新页面
- 配图生成中刷新页面
- 人为制造一次非法 JSON 输出，验证重试逻辑

## 本阶段边界

- 不处理 Go / Python 版本后端对齐
- 不引入向量库、RAG 或复杂长期记忆
- 不调整支付、统计、VIP 业务规则
- 不在本阶段新增 Reviewer / Refiner / Planner Agent

## 与版本管理方案的配合

第一阶段建议使用以下版本管理方式：

- 分支：`阶段/01-结构化输出`
- 阶段完成标签：`阶段-01-结构化输出完成`
- 阶段文档：本文件作为第一阶段主说明，阶段结束时补充实际完成结果与验证记录
