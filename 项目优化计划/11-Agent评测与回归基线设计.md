# Agent 评测与回归基线设计

## 文档目标

这份文档用于定义项目第四阶段后的下一条主线：建立一套可持续使用的 Agent 评测与回归基线。

目标不是一次性做复杂评测平台，而是先解决当前最现实的问题：

- 改了 Prompt，不知道是否真的更好
- 换了模型参数，只能凭感觉判断
- 加了长期记忆、路由、Reviewer 后，缺少稳定回归手段
- 节点失败、图片降级、重放能力虽然已有，但缺少统一验证入口

因此，第一版评测体系强调：轻量、可落地、可复用、能支撑后续 Agent 迭代。

## 为什么现在做

当前项目已经具备以下基础：

- Prompt / 模型 / 参数配置化
- 节点级执行观测
- 节点重放与失败节点重试
- 任务快照与任务恢复
- 长期记忆第一版
- 图片策略路由与失败降级链路

这意味着系统已经从“能跑起来”进入“需要稳定迭代”的阶段。如果没有评测基线，后续每次优化都只能依赖人工体感，风险会越来越高。

## 第一版设计原则

### 1. 先做最小闭环，不做复杂平台

第一版不引入额外数据库、不做独立管理后台、不做复杂评分模型。先用固定样本、规则统计和结果落盘的方式跑通。

### 2. 优先验证核心链路

优先覆盖以下能力：

- 标题生成
- 大纲生成
- 正文生成
- 正文 Reviewer
- 图片策略路由
- 图片需求分析
- 图片生成与降级

### 3. 结果必须可解释

评测不能只给一个“好 / 坏”结论，还需要能追溯：

- 使用了什么 Prompt 与模型参数
- 哪个节点失败
- 是否触发降级
- 是否命中长期记忆
- 总耗时与关键节点耗时如何

## 第一版评测范围

### 样本维度

建议准备一组固定样本，至少覆盖：

- 科技风格
- 情感风格
- 教育风格
- 轻松幽默风格

并在主题上覆盖：

- 纯文字表达型主题
- 适合流程图 / 架构图的主题
- 适合写实配图的主题
- 容易触发图片降级的边界主题

### 任务类型

第一版优先评三类任务：

1. 标准创作任务
   - 正常生成标题、大纲、正文、配图
2. 边界任务
   - 描述模糊、图片需求不明显、风格与主题组合较怪
3. 回归任务
   - 过去真实失败过、修复过的任务样本

## 评测指标设计

### A. 成功性指标

- 任务成功率
- 节点成功率
- 节点失败分布
- 失败节点重试成功率
- 图片生成成功率
- 图片降级触发率

### B. 时延指标

- 总任务耗时
- 各节点耗时
- 最慢节点分布
- 图片链路平均耗时

### C. 质量指标

第一版先不追求复杂自动评分，优先做可实现的轻量指标：

- 标题数量是否符合预期
- 大纲结构是否完整
- 正文长度是否达标
- 正文是否产生 Reviewer 修订
- 图片需求数量是否合理
- 最终图片数量是否匹配需求数量

### D. 决策与记忆指标

- 是否命中长期记忆
- 命中记忆的类型：
  - 用户偏好
  - 成功案例
  - 失败案例
- 图片策略路由来源分布
- Reviewer 决策来源分布

## 第一版样本结构建议

建议新增一组评测样本文件，例如：

```text
eval/
  cases/
    basic-tech.json
    emotional-story.json
    education-topic.json
    diagram-topic.json
    regression-image-fallback.json
```

单个样本建议至少包含：

```json
{
  "caseId": "basic-tech-001",
  "topic": "AI Agent 在内容生产中的应用",
  "style": "科技风格",
  "userDescription": "面向初学者，要求结构清晰，适合配图",
  "enabledImageMethods": ["PEXELS", "MERMAID", "SVG_DIAGRAM"],
  "tags": ["basic", "tech", "diagram"],
  "expectations": {
    "minTitleCount": 3,
    "minOutlineSections": 3,
    "minImageCount": 1
  }
}
```

## 第一版执行方式

### 方式一：本地命令执行

增加一个简单的评测入口，支持批量读取样本并执行。

执行目标：

- 逐个创建任务
- 等待任务完成或失败
- 拉取执行日志、节点快照、任务记忆
- 汇总输出评测结果

### 方式二：按阶段拆分执行

如果全链路批跑成本过高，第一版也可先支持：

- 标题 / 大纲 / 正文单独跑
- 图片链路单独跑
- 指定 case 进行 targeted replay 验证

## 结果输出建议

第一版结果先落到本地文件，建议格式：

```text
eval/
  runs/
    2026-05-25-001/
      summary.json
      cases/
        basic-tech-001.json
        emotional-story-001.json
```

其中：

- `summary.json`：记录总体成功率、平均耗时、失败节点分布
- 每个 case 文件：记录该样本的任务结果、节点结果、记忆命中、图片降级情况

## 与现有能力的复用关系

这套评测第一版不从零做，直接复用当前已有能力：

- 复用 `AgentExecutionStats` 获取节点耗时与状态
- 复用 `/article/node-replay/{taskId}` 获取节点快照
- 复用 `/article/memory-context/{taskId}` 获取记忆上下文
- 复用 `/article/retry-node` 做失败节点回归验证
- 复用现有任务快照与任务恢复链路

这能显著降低评测体系建设成本。

## 第一版不做的内容

为了控制范围，以下内容暂不纳入第一版：

- 向量检索评测
- 多模型横向大规模 Benchmark
- 复杂可视化报表平台
- 自动化 LLM 评分闭环
- 数据库存储全部评测记录
- CI 全自动跑全量长链路评测

## 推荐实施顺序

### 第一步：定义样本格式与最小样本集

先准备 5 ~ 8 个固定样本，覆盖标准、边界和回归场景。

当前状态：

- 已完成第一版 6 个固定样本落地

### 第二步：实现本地评测执行入口

能够跑批、收集结果、落本地 JSON。

### 第三步：补 summary 汇总逻辑

至少输出：

- 总成功率
- 节点失败分布
- 平均耗时
- 图片降级次数
- 记忆命中情况

### 第四步：加入最小回归操作手册

当修改以下内容时，必须至少跑一次最小回归：

- Prompt 模板
- 模型参数
- 正文 Reviewer 逻辑
- 图片策略路由逻辑
- 长期记忆召回逻辑

## 第一版验收标准

满足以下条件即可认为第一版落地：

- 有固定样本集
- 能本地批量运行
- 能输出每个 case 的任务结果
- 能汇总总体成功率和失败节点分布
- 能看到图片降级和长期记忆命中信息
- 文档中明确“什么修改后必须做回归”

## 下一步代码任务

文档完成后，建议立刻进入以下开发顺序：

1. 新增 `eval/cases/` 最小样本集
2. 增加本地评测执行入口
3. 增加 `summary.json` 汇总输出
4. 用 1 ~ 2 轮真实样本跑通并记录结果

## 阶段结论

阶段 5 的重点，不是再多做一个 Agent，而是让整个 Agent 系统进入“可验证、可回归、可安全迭代”的状态。

这一步一旦补齐，后续继续改 Prompt、改 Reviewer、改记忆、改图片路由，都会更稳。
