# 第二阶段实施方案

## 阶段目标

第二阶段聚焦“执行内核收敛”，目标不是继续堆功能，而是把当前 Java 主后端里并行存在的多套文章生成执行方式整理成一条主链路。完成后，项目应具备更清晰的编排入口、更一致的节点日志和更稳定的失败定位能力。

本阶段完成后，应达到以下效果：

- 文章生成主流程只保留一套明确的多 Agent 编排入口
- 节点执行、状态推进、SSE 推送、快照更新走统一通道
- 前后端能够基于统一阶段模型理解任务进度
- 后续接入记忆、路由、评审 Agent 时，不再需要重复改主流程

## 实施内容

### 1. 收敛文章生成主入口

围绕文章生成主链路，明确一个统一 orchestrator 作为唯一编排入口，其他旧式调用逻辑逐步下沉为可复用节点能力或兼容适配层。

本阶段重点：

- 统一“创建任务 -> 执行节点 -> 更新状态 -> 推送事件 -> 落快照”的调用路径
- 避免控制器、Service、异步任务里重复拼装流程
- 保留兼容层，但不再继续扩散旧流程分支

### 2. 标准化节点执行模型

为标题、大纲、正文、配图分析、图片生成、图文合成等步骤建立统一节点抽象。

建议每个节点至少具备：

- 节点名称与阶段标识
- 输入上下文与输出结果
- 标准日志记录
- 标准异常抛出
- 节点级耗时统计

这样后续增加 Reviewer、Planner、Memory 节点时，可以直接插入编排链路，而不是再写一套特殊流程。

### 3. 统一运行日志与观测字段

本阶段要让“为什么失败、失败在第几步、当前跑到哪一步”更容易看出来。

建议统一记录：

- `taskId`
- `phase`
- `node`
- `status`
- `message`
- `elapsedMs`
- `timestamp`

日志输出、SSE 事件、快照更新尽量复用同一份阶段语义，避免前后端各自维护不同叫法。

### 4. 清理双轨职责边界

对于当前并存的旧模式与新模式，本阶段不要求一次删除所有旧代码，但要先完成职责收口：

- 哪个类负责任务编排
- 哪个类负责节点执行
- 哪个类负责状态持久化 / 快照
- 哪个类负责 SSE 推送
- 哪个类负责结构化输出与工具调用

至少要做到：新增功能只接在统一主链路上，不再向旧分支继续增加逻辑。

## 接口与数据变化

第二阶段以内部重构为主，尽量减少对现有前端接口的破坏。

预计变化：

- 文章生成相关 Service 接口命名与职责会更清晰
- SSE `type / phase / payload` 结构保持兼容，但补充更稳定的节点信息
- 快照模型可增加节点级执行记录或最近事件摘要
- 如需新增调试接口，应仅用于开发态，不污染正式主流程接口

## 验收标准

### 后端验收

- 同一篇文章任务只经过一条明确的主编排链路
- 节点失败时能定位具体节点、阶段和错误信息
- 快照、SSE、日志中的阶段语义保持一致
- 旧流程即使暂时保留，也不再承担主执行职责

### 前端联调验收

- 创建文章主流程仍可正常使用
- 标题确认、大纲确认、正文生成、配图生成的阶段展示不回退
- 出错时前端能拿到更明确的失败节点或失败描述

### 手工场景

- 正常生成一篇完整文章
- 人为让标题 / 大纲节点失败一次，确认日志与快照可定位
- 正文生成中断后，确认恢复信息仍然正确
- 配图节点失败时，确认不会把整个任务状态写乱

## 本阶段边界

- 不在本阶段引入长期记忆、向量检索或用户画像系统
- 不在本阶段增加新的复杂 Agent 角色，只先整理现有执行骨架
- 不大改前端页面视觉，前端只做与执行内核对齐所需的最小调整
- 不处理 Go / Python 版本后端的架构同步

## 与版本管理方案的配合

第二阶段建议使用以下版本管理方式：

- 分支：`阶段/02-执行内核收敛`
- 阶段完成标签：`阶段-02-执行内核收敛完成`
- 推荐提交示例：
  - `重构：统一文章生成编排入口`
  - `重构：抽取文章节点执行上下文`
  - `功能：补充节点级运行日志`
  - `文档：更新第二阶段实施方案`
