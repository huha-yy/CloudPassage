# 第三阶段记忆架构设计

## 文档目标

这份文档用于明确第三阶段“记忆底座”应该怎么设计，重点回答四个问题：

- 记忆分几层
- 每层存什么
- 在什么时机写入和读取
- 先落到哪里，后面怎么扩展

本设计优先贴合当前项目现状，不直接引入复杂向量库或外部记忆平台，先把“可用、可读、可调试”的结构化记忆做好。

## 现有约束

结合当前代码，记忆设计需要遵守以下约束：

- 运行态主对象已经存在：`ArticleState`
- 任务快照已经存在：`ArticleTaskSnapshotServiceImpl`
- 节点观测已经存在：`ArticleNodeLogServiceImpl`
- 当前 Redis 已经承担任务快照和节点日志存储
- 用户模型目前只有基础字段，尚无偏好记忆字段
- 当前恢复与重试能力已经建立，记忆不能破坏现有恢复链路

因此，第三阶段记忆应优先复用：

- 现有 `taskId`
- 现有 `userId`
- 现有阶段枚举与节点命名
- 现有 Redis 存储方式

## 记忆分层

### 1. 任务级记忆

作用：服务“当前这一次创作”。

建议保存：

- 基础上下文：`taskId`、`userId`、`topic`、`style`、`userDescription`
- 过程决策：标题选择、大纲确认、图片策略、工具路由结果
- 执行摘要：当前阶段、最近成功节点、最近失败节点、重试次数
- 内容摘要：最终标题、大纲摘要、正文摘要、图片需求摘要
- 质量信号：是否发生结构化输出失败、是否触发降级、是否人工介入

这个层级优先用于：

- 恢复执行
- 节点重试
- 后续 Planner / Reviewer 读取上下文

### 2. 用户级偏好记忆

作用：服务“同一个用户之后的创作”。

建议保存：

- 常用文章风格
- 常用图片方式组合
- 常用标题偏好
- 最近几次人工修改倾向
- 常见失败模式与规避建议

这个层级优先用于：

- 新任务初始化默认值
- 标题生成前提示
- 图片工具路由前策略补充

### 3. 系统级策略记忆

第三阶段先不单独实现为复杂模块，但设计上预留。

建议未来承载：

- 工具成功率统计
- 常见失败节点与推荐降级路径
- 不同题材下的默认工具组合

## 数据结构建议

### 任务级记忆结构

建议新增独立模型，例如 `ArticleTaskMemoryVO`，不要直接塞进 `ArticleState` 全部字段。

建议字段：

- `taskId`
- `userId`
- `topic`
- `style`
- `userDescription`
- `currentPhase`
- `lastSuccessNode`
- `lastFailedNode`
- `retryCount`
- `selectedTitle`
- `outlineSummary`
- `contentSummary`
- `imageStrategy`
- `toolDecisions`
- `qualitySignals`
- `manualActions`
- `updatedAt`

说明：

- `ArticleState` 继续作为运行态对象
- `ArticleTaskSnapshotVO` 继续负责恢复主链路
- `ArticleTaskMemoryVO` 负责更长期、更抽象的任务经验沉淀

### 用户级偏好结构

建议新增独立模型，例如 `UserCreationPreferenceVO`。

建议字段：

- `userId`
- `preferredStyles`
- `preferredImageMethods`
- `preferredTitlePatterns`
- `outlineEditPatterns`
- `contentToneHints`
- `recentFailureTags`
- `updatedAt`

后续如果需要落库，可再拆成实体或 JSON 字段。

## 存储策略

### 第一阶段落地策略

先采用“Redis + 轻量持久化预留”的方案。

建议：

- 任务级记忆先放 Redis
- 用户级偏好先放 Redis，后续再评估是否落 MySQL

原因：

- 当前任务快照和节点日志已经在 Redis
- 读取路径短，接入速度快
- 方便先验证记忆字段是否真能影响执行

建议 Redis Key：

- `article:task:memory:{taskId}`
- `user:creation:preference:{userId}`

### 第二步演进策略

当字段稳定后，再考虑：

- 用户偏好记忆落 MySQL
- 任务记忆保留 Redis 热数据 + 完成后摘要入库

这样可以兼顾：

- 恢复速度
- 调试便利
- 长期留存

## 读写时机

### 任务级记忆写入时机

- 创建任务后：写入基础上下文
- 标题确认后：写入最终标题选择
- 大纲确认后：写入结构摘要与人工修改痕迹
- 图片策略确定后：写入工具路由决策
- 节点失败后：写入失败节点、错误标签、重试次数
- 任务完成后：写入最终内容摘要与执行结果摘要

### 任务级记忆读取时机

- 任务恢复时
- 节点重试时
- Planner 决策前
- Reviewer 分析前

### 用户级偏好写入时机

- 用户完成文章后
- 用户多次重复选择同类风格或配图方式后
- 用户人工修改大纲或标题后

### 用户级偏好读取时机

- 新建文章任务时
- 标题生成前
- 图片方式推荐前

## 与现有代码的衔接方式

### 1. 不替换快照，只补一层记忆

当前：

- `ArticleTaskSnapshotVO` 负责“恢复现场”

第三阶段新增：

- `ArticleTaskMemoryVO` 负责“沉淀经验”

两者职责不要混在一起。

### 2. 先在服务层增加记忆门面

建议新增统一门面，例如：

- `ArticleMemoryService`

首批方法建议：

- `saveTaskMemory(...)`
- `getTaskMemory(taskId)`
- `saveUserPreference(...)`
- `getUserPreference(userId)`
- `buildCreationMemoryContext(taskId, userId)`

### 3. 记忆上下文不要直接暴露原始 Redis 结构

建议对 Agent 使用统一上下文对象，例如：

- `ArticleMemoryContext`

里面聚合：

- 当前任务记忆
- 当前用户偏好
- 最近失败节点摘要
- 工具路由建议

这样后续接 Planner / Reviewer 时不会重复拼装。

## 推荐实施顺序

### 第一步：任务级记忆最小版

- 新增任务记忆模型
- 接入 Redis 存储
- 在标题确认、大纲确认、失败、完成四个时机写入

### 第二步：用户级偏好最小版

- 新增用户偏好模型
- 在任务完成后从结果反推偏好并写入
- 在创建任务时读取并作为默认建议

### 第三步：记忆上下文接入执行链路

- 在标题生成前读取偏好
- 在图片策略路由前读取任务记忆和用户偏好
- 在失败节点重试时读取最近失败摘要

## 验收标准

- 至少一类任务记忆可被恢复链路复用
- 至少一类用户偏好可影响新任务默认策略
- 记忆模型与任务快照、节点日志职责边界清晰
- Redis Key、字段和 TTL 规则明确
- 前后端主链路不因记忆接入而被破坏

## 当前结论

第三阶段的记忆底座不应从复杂技术栈开始，而应从现有 `ArticleState + 任务快照 + 节点日志 + Redis` 之上再补一层“结构化经验记忆”。先把任务级记忆和用户级偏好记忆做出来，后面的 Planner、Reviewer、工具路由和真正单节点重放才有统一上下文可用。
