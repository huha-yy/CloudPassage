# 项目升级阶段版本管理方案

## 概要

本次项目升级采用“主分支稳定 + 阶段分支开发 + 小步提交 + 阶段打标签 + 阶段文档同步”的版本管理策略。目标是：

- 每个阶段都能独立开发、独立验收、独立回滚
- `main` 分支尽量始终保持可运行、可回退
- Git 历史能清晰体现“这个阶段做了什么”
- 后续你自己复盘、继续开发、展示项目时都更容易阅读

默认规则已经确定：

- 分支方式：每个阶段一个分支
- 提交方式：阶段内多个小提交
- 阶段留痕：每阶段完成后打标签
- 文档节奏：每阶段完成后更新一次阶段说明
- 命名与描述：尽量使用中文

## 版本管理规则

### 1. 分支规则

- `main`：稳定主分支，只接收“一个阶段完成并验收通过”的代码，不直接在上面做升级开发。
- 每个升级阶段新建一个阶段分支，推荐命名：
  - `阶段/01-结构化输出`
  - `阶段/02-任务快照恢复`
  - `阶段/03-SSE协议统一`
  - `阶段/04-创作页状态重构`
  - `阶段/05-执行内核收敛`
- 如果某阶段过大，可在阶段分支下继续拆临时子分支，但最终都回到该阶段分支，不直接进 `main`。

### 2. 提交规则

- 每次提交只表达一个清晰意图，不混杂多个主题。
- 提交信息尽量使用中文，推荐格式：
  - `功能：新增结构化输出执行器`
  - `重构：统一文章生成SSE消息结构`
  - `修复：处理大纲JSON解析失败重试`
  - `测试：补充任务快照恢复场景`
  - `文档：更新第一阶段升级说明`
- 一个阶段内允许多个提交，但每个提交必须是“可理解、可回看、可定位”的。

### 3. 合并与阶段完成规则

- 一个阶段只有在以下条件同时满足时才允许合并到 `main`：
  - 核心功能已经完成
  - 本阶段自测通过
  - 关键流程手工验证通过
  - 没有已知阻断问题
  - 阶段说明已更新
- 合并后立即打阶段标签，推荐命名：
  - `阶段-01-结构化输出完成`
  - `阶段-02-任务恢复完成`
  - `阶段-03-SSE统一完成`
- 标签说明中写清：本阶段目标、完成内容、验证方式、回滚点说明。

### 4. 阶段文档规则

- 每个阶段结束后更新一次阶段文档，不要求每次提交都更新。
- 阶段文档至少包含：
  - 阶段目标
  - 本阶段改动摘要
  - 涉及模块
  - 验证结果
  - 已知限制
  - 回滚方法
- 建议统一放在一个中文目录中，例如：
  - `项目优化计划/阶段01-结构化输出.md`
  - `项目优化计划/阶段02-任务恢复.md`

## 实施节奏

### 第一阶段开始前

- 从当前稳定的 `main` 创建 `阶段/01-结构化输出`
- 保留现有未提交内容，先确认哪些属于升级文档、哪些属于界面文案调整，再决定是否单独整理

### 每个阶段内部

- 先完成核心主线，再补测试，再补文档
- 阶段内如果出现临时修复，仍提交到当前阶段分支，不插到别的阶段里
- 如果发现需求已经超出本阶段目标，不继续叠加，单独记入下一阶段计划

### 每个阶段收口

- 运行测试或构建检查
- 手工验收核心链路
- 更新阶段说明
- 合并回 `main`
- 打阶段标签
- 记录下一阶段起点

## 测试与验收要求

- 每个阶段至少要有一份“本阶段验收清单”，写明必须验证的场景。
- 文章主链路相关阶段，至少覆盖：
  - 正常创建文章
  - 阶段切换是否正确
  - 页面刷新后是否符合预期
  - 失败场景是否可见、可定位
- 如果阶段只涉及后端，也要至少验证前端主链路不被破坏。
- 如果阶段涉及协议变更，必须验证旧页面行为是否仍可用，或者明确记录“不兼容变更”。

## 默认约定

- 中文优先：分支名、标签名、提交说明、阶段文档都尽量使用中文，必要时可混用少量英文技术词。
- `main` 不做日常开发，只作为阶段成果汇总分支。
- 不做“所有升级都堆在一个超长分支里”的方式。
- 不做“全部完成后再统一整理历史”的方式，避免后续难以回溯。
- 如果某阶段合并后发现问题，优先基于标签定位并回滚，不在 `main` 上继续堆补丁掩盖问题。
